# MIRAGE — Platform Principles
*Established May 15, 2026. Update as decisions compound.*
*This is not a spec. It's the thinking that specs get built from.*

---

## The Character Defines the System

Every character problem is a system requirement in disguise.

Sloan's 964 isn't a detail to tweak — it's proof that the visual canon system has to hold specific objects consistently, not just spaces. Her context bleed problem is proof that ownership tagging in synthesis has to be airtight. Her performing vs being problem is proof that the system can't rely on rules to produce genuine voice.

When Vera gets built, she'll have her own version of all of those. Different details, same underlying requirements. The system either handles them or it doesn't. Sloan is how we find out.

Every fix on Sloan should be asked: is this fix generic enough that any Icon benefits from it automatically? If yes, it's platform work. If it's specific to her world, it goes in her Persona Builder fields — which is exactly where another Icon would put their equivalent details.

**Fix the system. The character defines what the system needs.**

---

## One Track, Not Two

Sloan is the proof of concept. Every problem solved with her, every field added to her Persona Builder, every behavioral fix that holds — that becomes the template. Other Icons inherit a working system, not a broken one.

There is only one track right now. Get Sloan right. Everything else follows.

The parallel build doesn't start until Sloan is consistent. Not just technically functional — consistent. Visual canon holding. Voice producing something genuinely interesting unprompted. World coherence not breaking under real conversation pressure.

When those pass on Sloan, the platform is proven in the hardest case. Then it gets abstracted and built clean for everyone else. That build will go faster because the hard problems were already solved.

**A spec requires knowing what actually works. The spec gets written when Sloan is consistent.**

---

## Persona Builder is the Template

Every field added to Sloan's Persona Builder is a field that exists for every future Icon.

The Persona Builder isn't a Sloan tool. It's the Icon creation system. Sloan just happens to be the first Icon built on it. When a new Icon gets created, she fills in the same fields with her own details. The system handles the rest.

This means every Persona Builder addition needs to be thought of as a platform decision. Not "what does Sloan need" but "what does every Icon need."

**The constraint is never the system. It's only ever imagination.**

---

## Icon Agents

At scale, manual oversight breaks. One Sloan is manageable. One hundred Icons isn't.

Each Icon needs an agent that owns her. The agent:
- Watches her conversations continuously
- Flags drift before it compounds
- Catches Oliver Framework violations before they ship
- Monitors image quality and visual canon consistency
- Tracks content she's generating and whether it's consistent with who she is
- Reports to Andie

Andie escalates to Nate only when it actually needs him.

The agent doesn't just monitor — it accumulates. Every flag, every pattern, every drift correction becomes institutional knowledge that doesn't disappear when a chat retires. The character stays consistent because something is always watching.

**This is a core platform requirement, not a future feature. Build it before the second Icon.**

---

## The Repetition Problem

Decisions made in one chat get lost when that chat retires. The same ground gets re-explained to fresh instances. This is a system failure, not a people failure.

The fix is not better note-taking. The fix is an architecture where decisions compound instead of evaporate.

MIRAGE_STATE.md is part of that architecture. This document is part of that architecture. The Icon agent is part of that architecture.

Every strategic decision made in any chat goes somewhere permanent before the session ends. Not optional.

---

## What Sloan Has to Pass Before Anything Else Gets Built

1. Visual canon holds across 10 consecutive images — feels like the same place every time
2. Voice produces something genuinely interesting unprompted — not technically correct and empty
3. World coherence doesn't break under real conversation pressure — multiple subscribers, multiple sessions
4. Layer 2 pre-send review is live — hallucinated images cannot reach real people
5. Context bleed is fixed — her world is hers, subscriber's world is theirs

When all five pass consistently — not once, consistently — the template is ready.

---

## The Live System Problem

The Task Checklist and Project Dashboard exist but they're passive. They only reflect reality when someone manually updates them. Nobody does consistently. So they drift and become useless.

What's needed is a system that writes itself.

When a fix gets deployed — the checklist updates. When drift gets flagged by an agent — it's logged automatically. When a new Persona Builder field gets added — the template updates. When a decision gets made in any chat — it lands somewhere permanent without anyone having to remember to write it down.

The state of the platform should always be current because the platform maintains its own state. The manual update problem goes away when the system reports on itself.

This is also what Andie is for. Not just escalating Icon agent flags — maintaining the living record of what's working, what's broken, what got decided, and what changed. So Nate never has to repeat himself because Andie already knows.

---

## Institutional Memory

Two months of the most important thinking about what Mirage is, what an Icon needs to be, what the system has to handle — it lives in retired chats that nobody reads. The decisions that shaped everything are the least recoverable things in the system.

Code gets committed. Bugs get fixed. Conversations disappear.

What needs to exist is something that captures strategic thinking in real time. Not a summary written at the end of a session. Not a state doc that requires manual updating. Something that recognizes when a conversation crosses from tactical to important — a principle gets established, a direction changes, a hard problem gets solved — and writes it somewhere permanent automatically.

Andie is the right vehicle for this. She monitors every chat. When something important gets said she knows it's important and she captures it. The platform principles doc stops being something GZA writes at the end of a session and becomes something that writes itself from every session across every chat.

The character defines the system. One track not two. Agents owning Icons. The system reporting on itself. None of that should require Nate to re-explain it to the next chat instance. Andie already has it.

That's what makes the whole operation sustainable at scale. Not better note-taking. Institutional memory that accumulates automatically and never gets lost.
